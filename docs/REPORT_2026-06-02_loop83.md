## Summary

**Cross-layer traceability loop -- new theme (not the SSOT/codegen arc of 79-82). A format is "Tier-1 / on-die" in THREE independent places: the ROM metadata (`CATALOG` records with `FLAG_ON_DIE`), the actual silicon decode path (the `case (fmt_id_r)` arms in `tt_um_trinity_corona.v` that set `has_decoder=1`), and the test coverage (`fmt_ids` exercised via `send_cmd` in `test_decoders.py`). These three sets MUST be identical -- otherwise the chip either misreports its own capabilities (ROM says on-die, no decoder) or ships an untested/undeclared decode path. Nothing checked this. The sets agree today (22 each), so the new `test/test_ondie_coverage_crosscheck.py` locks it in. It also verifies `FMT_<name>=id` naming agrees between the top module and the test, and that every mux arm maps to a defined localparam. All PASS; a mutation (drop `FLAG_ON_DIE` from bf16) is caught (rc=1, reports mux-only/tested-only=[8]). Wired into CI. Pure stdlib.**

## The gap (traceability terms)

Requirements-traceability practice forbids "orphans" (implementation with no requirement) and "childless parents" (requirement with no implementation). Mapped to Corona:

| Drift | Traceability term | Consequence |
| --- | --- | --- |
| `FLAG_ON_DIE` set but no mux decoder | childless parent | ROM advertises a decode the silicon can't do |
| mux decoder but `FLAG_ON_DIE` clear | orphan | undeclared decode path; ROM under-reports the chip |
| on-die format with no cocotb test | untested feature | decode path ships unverified |
| `FMT_BF16` differs between top module and test | silent renumber | test exercises a different fmt than the hardware routes |

Loops 79-82 pinned the ROM *data path* (SSOT -> packer -> emitted RTL -> GLS golden). This loop pins the orthogonal *capability/coverage* axis (ROM self-description <-> decode hardware <-> tests).

## Research (this loop, web available)

The work maps directly onto **requirements-to-implementation traceability** and **self-describing capability registers**:

| Source | Finding | Bearing on Corona |
| --- | --- | --- |
| Traceability practice (no orphans / no childless parents) | every feature flag must trace to an implementation and a test; gaps are exactly orphans/childless parents | This test is a bidirectional trace matrix for the on-die capability bit |
| Self-describing capability registers (USPTO 7,096,440; 8,954,541) | systems should validate that a device's *advertised* capabilities match the *physical* implementation, ideally at power-on (POST) | Next step: a power-on self-test on silicon that reads each `FLAG_ON_DIE` bit and exercises its decode path |
| SVA / formal capability binding | bind an assertion that "capability bit set <=> functional path exercisable" | A formal version of this set-equality check |

Sources: [requirements traceability](https://www.trace.space/blog/what-is-requirements-traceability), [spec-to-hardware verification (USPTO 7,096,440)](https://image-ppubs.uspto.gov/dirsearch-public/print/downloadPdf/7096440), [self-describing modules (USPTO 8,954,541)](https://image-ppubs.uspto.gov/dirsearch-public/print/downloadPdf/8954541).

## Three Collaboration Options for the Next Loop

### Option A: Power-on self-test (POST) on silicon
Extend `post_silicon/test_corona.py` so the demoboard reads each `FLAG_ON_DIE`
bit from the ROM and then exercises that format's decode path, failing fast on
any advertised-but-broken decoder. Turns this static trace into a runtime POST
(per the self-describing-capability-register literature).

### Option B: Auto-generate the FMT_ constants from the SSOT
`FMT_<name>=id` is hand-maintained in BOTH `tt_um_trinity_corona.v` and
`test_decoders.py`. Emit a single `fmt_ids.vh` + Python module from the t27
catalog so the mux, the tests, and the ROM share one fmt_id source -- removing
the renumber hazard this test currently only *detects*.

### Option C: Formal capability assertion
Add an SVA/SymbiYosys property: for every fmt_id, `has_decoder` reads 1 iff the
ROM record for that fmt_id has `FLAG_ON_DIE` set. Proves ROM<->mux agreement
over all 128 fmt_id values, not just the 22 enumerated on-die ones.
