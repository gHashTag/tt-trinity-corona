# Loop 126 Report -- 2026-06-03

## Summary

**Audited the gf16 COMPUTE-PATH primitives (executes the prior loop's Option C) --
the units actually instantiated on the shipped Phi/Euler/Gamma silicon -- to answer
"is there an active silicon arithmetic defect?" Answer: no. The two instantiated
primitives, `gf16_dot4` (the 4-MAC reduction tree used by the tiles/mesh) and
`tri_mant_mul` (the shift-add multiplier in `fbb_active_path`), are correct;
`gf16_dot4` inherits only the documented gf16 add/mul caveats with no new defect.
The real bug found was in `gf16_to_fp16`: verilator confirmed an inferred LATCH on
`fp_out` (every |value|<1.0 was routed to a "subnormal" branch that never assigned
the output) plus a wrap-prone 5-bit overflow add -- but this converter is NOT
instantiated on any die's silicon top (standalone, never in a GDS netlist), so it
is dead-code. Fixed it anyway (correct sub-1.0 -> FP16-normal mapping, fp_out
assigned on every path), verified EXHAUSTIVELY over all 65536 GF16 inputs, and
ported the verified fix to all three dies. `gf16_to_posit16` is a known simplified
stub (left as-is). Net: the shipped compute path is confirmed clean, and the one
real primitive bug is fixed line-wide.**

## Why

The prior loop's cross-die audit showed the buggy gfN_add units are off the compute
path. The natural follow-up -- and the only place a latent bug could be ACTIVE on
silicon -- is the gf16 primitives that the tops actually instantiate. Confirming
those are correct is the reassuring, decision-relevant conclusion; finding and
fixing the converter latch is the concrete by-product.

## Implementation

- **Audit (read-only first):** the primitives are byte-identical across all three
  dies. `gf16_dot4` = 4x`gf16_mul` + 3x`gf16_add` (correct tree); `tri_mant_mul` =
  partial-product sum (exact). Confirmed via instantiation search that
  `gf16_to_fp16` / `gf16_to_posit16` are standalone (not on any `tt_um_*` top).
- **Fix (`tt-trinity-{gamma,phi,euler}/src/gf16_to_fp16.v`):** rewrote the normal
  path to compute the FP16 biased exponent directly (`gf_exp-16`, signed) and assign
  `fp_out` on every branch -- overflow->Inf, below-min-normal->signed zero, else
  normal. Defaulted `fp16_to_gf16`'s intermediate regs to kill its latches too.
- **Verify (`tt-trinity-gamma/test/gf16_to_fp16_test.py`):** all 65536 GF16 inputs
  vs a reference; 65536/65536 pass; verilator `-Wall` clean (bar the benign
  two-modules-per-file note). Ported the exact verified file to phi/euler.
- **Doc:** `docs/CROSS_DIE_GF_AUDIT.md` extended with the compute-path verdict table.

## Research (this loop, web available)

The dead-code-vs-live distinction is again the crux: an inferred latch is a real
synthesis hazard (it would put a transparent latch in the netlist and create timing
/ X-propagation problems), but only if the module is in the build -- here it is not,
so the bug is a latent library defect, not a silicon fault. The fix itself is a
textbook narrow-float conversion point: the original conflated "negative unbiased
exponent" with "subnormal", but values in [2^-14, 1) are ordinary FP16 normals;
only magnitudes below the destination's min normal are subnormal/underflow
(Goldberg; IEEE-754 binary16). Assigning every output on every path (defaulting at
the top of the always block) is the standard latch-avoidance discipline.
