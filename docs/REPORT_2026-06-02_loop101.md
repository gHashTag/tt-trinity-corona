## Why (the Loop 92 bug class, eliminated)

Loop 92 found 5 FP8_E5M2 bring-up vectors that were hand-transcribed WRONG and
would have failed a correct chip. The CI guard (Loop 92-94) catches such errors,
but they could still be *authored*. Generating the vectors from the references
means the expected value for each input is computed by the same spec-derived
decoder the RTL was proven equal to -- so a transcription error is impossible by
construction, not merely caught after the fact.

## Research (this loop, web available)

This is the codegen-from-single-source pattern applied to the bring-up oracle:
the same principle as SystemRDL/PeakRDL (generate every view -- RTL, checker,
docs -- from one source so they cannot drift) and the regenerate-and-diff CI gate
for committed generated artifacts. The post-silicon literature (Loop 93) is
explicit that golden vectors should be derived from a model rather than hand-
authored, and that the strongest setup pairs model-derived vectors with an
INDEPENDENT reference cross-checking them. Corona now satisfies both: the bring-up
vectors are derived from the independent references, and `test_post_silicon_vectors`'s
separate mapping cross-checks the result.

Sources: [regenerate-and-diff / generated-file CI gate (git-diff)](https://git-scm.com/docs/git-diff),
[single-source-of-truth codegen (SystemRDL/PeakRDL)](https://www.accellera.org/activities/working-groups/systemrdl/systemrdl-details).
