## Why (defends against vacuous guards)

A guard that cannot fail gives false confidence -- the "oracle rot" failure named
in Loop 95's research. The standard defence is mutation testing: deliberately
inject a bug and confirm the test catches it. Loops 86-95 each demonstrated this
once, by hand. This loop institutionalises it: the discriminating power of the
verification suite is now itself verified on every CI run.

## Implementation

`test/test_mutation_guards.py` (stdlib + iverilog):
- **RTL mutations** (non-destructive): copy a decoder `.v` to a tempdir, flip one
  output literal, compile + exhaustively sweep with iverilog, and assert the
  independent reference (from Loops 86-90) disagrees on >=1 input. The real
  `src/rtl` is never modified. 4 decoders chosen to span families: LUT (nf4),
  power-of-two (e8m0), ternary (bitnet), variable-regime computed (posit8).
- **Codegen mutations** (monkeypatch, restored in `finally`): misplace a field in
  `gen_rom.pack_record` -> `test_ssot_layout_crosscheck.main()` returns nonzero;
  change a `CATALOG` mantissa -> `test_rom_emitted_golden.main()` returns nonzero.
- Outcome accounting: detected / missed / skipped. Any `missed` -> fail; RTL
  cases skip gracefully (not pass) without iverilog.

## Research (this loop, web available)

Mutation testing is the canonical "test of the tests": seed a fault (a mutant)
into the artifact and require the suite to kill it; surviving mutants reveal
weak or vacuous checks. It is the direct, constructive answer to Loop 95's
"who validates the validator?" -- rather than another oracle, you perturb the
system and confirm the existing oracle reacts. This complements the other layers
already in place: independent references (validated oracles), the completeness
gate (no oracle gaps), and now mutation testing (no vacuous oracles). The classic
caution -- equivalent mutants that change no observable output and thus can't be
killed -- is why this gate uses output-changing literals and includes a no-op
self-check to confirm it can report a miss.

Sources: [Mutation testing (Wikipedia)](https://en.wikipedia.org/wiki/Mutation_testing),
[The Often Overlooked Test Oracle (AST)](https://associationforsoftwaretesting.org/2023/01/10/the-often-overlooked-test-oracle/).
