# Loop 128 Report -- 2026-06-03

## Summary

**Triaged the rest of the full-design lint (executes the prior loop's Option A) and
found a SECOND active on-silicon bug. Of the 11 `WIDTHTRUNC` warnings from the
lint-from-top, 10 are benign Verilog-2005 "shift then mask to a narrow wire" idioms
(the mask guarantees the truncated high bits are zero -> lossless: the popcounts,
vsa_matmul_8x8/16x16, blake3_anchor, bitnet_encoder:57). The 11th is real:
`bitnet_encoder.v:92` -- `ternary_dot`'s `neuron_base` is `[9:0]` (10-bit) but
Layer 1 calls it with `k*64` (up to 1984) for the 32 hidden neurons, so it truncates
mod 1024 and ALL 16 upper neurons alias a lower one (`h1[k]==h1[k-16]`); the 64->32->8
demo encoder realizes only 16 distinct hidden neurons. `bitnet_encoder` is
instantiated on the Gamma and Euler silicon tops (not Phi). Because `w_gen(addr)` is
a pure function, identical `neuron_base` yields identical weights -> the aliasing is
deductively certain (confirmed by `tools/bitnet_neuron_base_check.py`, which reads
the live param width). Same root class as last loop's gf16_mul: a width one bit too
short -> wraparound. Severity is moderate/latent (synthetic demo weights, golden
vectors still match), but it is a real architectural defect on shipped silicon;
fix for a respin is a one-bit widening. Frozen sources left as taped out.**

## Why

The prior loop's gf16_mul find proved the full-design lint is productive on live
logic, so triaging the remaining flags was the natural completion -- separating real
defects from benign idioms is exactly the value of the audit. Doing it per-site (not
bulk-silencing) is what surfaced the bitnet bug among ten false alarms.

## Implementation

- **Triage:** read each of the 11 `WIDTHTRUNC` sites. 10 are mask-extraction idioms
  (`(wide >> shift) & narrow_mask` assigned to a narrow wire) -- safe because the
  mask zeroes everything above the target width.
- **Bug:** `bitnet_encoder.v` `neuron_base [9:0]` vs `k*64 <= 1984`. Built the exact
  collision map (k=16..31 -> 0..960, each == k-16). Confirmed `bitnet_encoder` is on
  Gamma's `tt_um_trinity_max_true` and Euler's `tt_um_ghtag_trinity_gf16` tops.
- **Evidence + docs:** `tools/bitnet_neuron_base_check.py` (reproduces the collision,
  reads the live RTL param width as a regression check) and
  `docs/FULL_DESIGN_LINT_TRIAGE.md` (full per-site table + the bitnet finding + fix).

## Research (this loop, web available)

The triage discipline is the point: a width-mismatch lint is high-recall, low-
precision -- most hits are deliberate masking, a few are wraparound bugs -- so
bulk-silencing it (a common reflex) would have buried the bitnet defect along with
the ten idioms. Reading each site and asking "does the mask make the truncation
lossless?" is the cheap discriminator. Both real bugs this arc (gf16_mul, bitnet)
are the same archetype: an intermediate one bit too narrow so a value that should
carry out wraps -- the classic off-by-one-bit that unit tests miss (the value rarely
lands exactly at the wrap) but static width analysis catches for free. The bitnet
case is also a reminder that "synthetic/demo" RTL still ships on the die and still
deserves correctness, even when no trained workload exposes it.
