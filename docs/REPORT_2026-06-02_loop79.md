## The gap (why this matters)

`specs/corona/rom_layout.t27` is the SSOT and declares itself FROZEN: any layout change "requires a PR update to t27, re-generation of the ROM, and a fresh CI run." But the freeze was only documented, never enforced. The field bit-positions lived independently in:

1. `rom_layout.t27` — `FIELD_*.{hi,lo,width}` (the SSOT)
2. `gen_rom.py` `pack_record()` — literal shifts (`<< 72`, `<< 68`, ...) and masks
3. `test_rom_spec_crosscheck.py` `extract_fields()` — a *third* set of literal shifts

An edit to any one would silently desync the others. The existing `test_rom_spec_crosscheck.py` checks ROM *data* against published numeric specs but never checks the *structure* against the SSOT.

## Implementation

`test/test_ssot_layout_crosscheck.py` (pure Python, no cocotb/sby; runs in <1s):

- **Field layout (functional):** parse `FIELD_*` from `rom_layout.t27`; feed `pack_record()` a probe with distinct per-field values; extract each field at the SSOT-declared `[lo, lo+width)` and assert it equals the probe input. Catches any shift/width drift. `phi_distance_q16` (computed internally) is checked via `gen_rom.phi_distance()`.
- **Width sum == RECORD_BITS == 80.**
- **ENCODING_\* (10)** vs `ENC_*`, **FLAG_\* (4)** vs `FLAG_*` (matched by bit-value, two flags renamed), **STATUS_\* (8)** vs `ST_*` (set == 0..7, anchors VERIFIED=0 / SPEC=7).
- **Cardinality:** TOTAL_FORMATS (oracle) == RECORD_COUNT (layout) == len(CATALOG) == 80; CLUSTER_COUNTS sum == 80.

CI: new step `SSOT<->codegen layout cross-check` after the existing ROM-spec step.

## Three Collaboration Options for the Next Loop

### Option A: Formal ROM miter (spec <-> generated RTL)
The ROM is finite (80 records x 80 bits = 6400 bits). Build a SymbiYosys/EQY miter that proves `format_rom.v` output == `gen_rom.py` packing for all 80 addresses, and that each unpacked field matches the SSOT layout — a closed-form proof replacing today's sampled checks. (Per FormalRTL / SpecLoop research.)

### Option B: Collapse the third layout copy
`test_rom_spec_crosscheck.py` `extract_fields()` still hardcodes shifts. Refactor it to import field offsets from a single helper that reads `rom_layout.t27`, so there is exactly ONE machine-readable layout source. Reduces drift surface from 3 copies to 1 + generated consumers.

### Option C: RDL/SSOT single-source spike
Prototype emitting the ROM packer itself from the `.t27` SSOT (or a SystemRDL transcription), so `pack_record` is generated rather than hand-written — eliminating the hand-coded packer entirely and making the "mechanical end-to-end" claim literally true at the layout level. (Per PeakRDL research.)
