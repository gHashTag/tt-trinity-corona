## Why this matters (oracle rot)

The post-silicon suite is the oracle that judges the physical chip. The testing
literature names the precise failure this arc has been fighting: **"oracle rot --
the oracle itself becomes outdated or wrong, and tests pass even when the code is
broken; especially dangerous with hardcoded expected values someone copy-pasted
without verifying."** That is exactly the Loop 92 FP8_E5M2 bug. The field's
remedy is to (a) treat the oracle with the same rigor as the code, (b) prefer
*validated* over *unvalidated* oracles, (c) use mutation testing to prove the
checks aren't vacuous, and (d) meta-test for oracle gaps (tests that assert
nothing meaningful). Loops 86-94 turned Corona's bring-up oracle from unvalidated
to validated; this loop adds the meta-test (d) so the validated property is
maintained, not just achieved once.

## Research (this loop, web available)

The test-oracle literature frames the whole 86-95 arc:
- **Oracle problem** -- determining the correct output for an input is formally
  hard; every assertion rests on an oracle that can itself be wrong.
- **Oracle rot / unvalidated masters** -- Doug Hoffman: most saved oracles are
  *unvalidated* ("we looked and didn't notice anything") vs *validated* ("we
  checked every value"). IBM found 30% of field bugs had been hit by tests but
  not noticed -- an oracle/observer gap, not a coverage gap.
- **"Who validates the validator?"** -- the answer is a layered defence: review
  oracles like code; use partial/property oracles where exact truth is unknown;
  use mutation testing as a direct test-of-the-tests; and meta-test for oracle
  gaps. This loop is the meta-test; the per-loop mutation checks are the
  mutation-testing layer; the independent references are the validated oracle.
