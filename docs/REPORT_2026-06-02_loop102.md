## Why this (and not more code)

Honest assessment after ~23 loops: the verification suite is comprehensive and
self-maintaining (ROM, dossier, and bring-up vectors all generated +
freshness-gated; 18 CI gates; mutation-tested). The genuine remaining weakness was
that this rigor -- a major asset for a shuttle submission and any write-up -- was
undocumented for a reader of the repo. Fixing that is low-risk and high-signal.
The carried "generate fmt_ids" option, by contrast, would touch cocotb test files
I cannot verify locally, for a bug class the Loop 85 consistency gate already
prevents; that risk/benefit did not justify acting this loop.

## Research (this loop, web available)

Verification documentation -- a concise, accurate summary of the evidence behind a
design's correctness -- is itself a recognised deliverable: assurance cases /
safety cases (ISO 26262 Part 2, GSN) argue a top claim ("the decoders are
correct") from structured evidence. Corona's five evidence layers + dossier are a
lightweight assurance case; this loop makes the top of that argument visible in
the README rather than buried across 100+ loop reports. The accompanying
discipline -- only claim numbers you can verify against the artifact -- is why the
audit checked every count rather than restating them.

Sources: [Assurance case / safety case (Wikipedia)](https://en.wikipedia.org/wiki/Assurance_case),
