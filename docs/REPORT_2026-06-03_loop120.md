# Loop 120 Report -- 2026-06-03

## Summary

**Fixed all five mul-broken GoldenFloat rungs in one pass (executes the prior
loop's Option B). gf20/24/32/64/128_mul shared a single bug: `mant_product` was
declared `2M` bits wide instead of `2*(M+1)`, truncating the leading bit of the
`(M+1)x(M+1)` significand product, with every normalize index 2 too low and a
rounding path that bumped the exponent without incrementing the mantissa --
products came out ~half magnitude (gf32: `1.0*1.0 -> 0.5`). All five were
regenerated by a new `test/gen_gf_mul_fix.py` using the verified gf16/gf8
algorithm (full-width product, correct normalize, guard/round/sticky
round-to-nearest), special-value constants preserved verbatim. Verified by the
exponent probe (all five flip OK) and a new `test/gf_mul_sweep.py` that runs 7200
finite-normal pairs/rung against an EXACT-RATIONAL reference (needed -- gf64/gf128
overflow IEEE double): max 0.500 ULP on every rung. GoldenFloat arithmetic is now
7/9 correct (gf8, gf16, gf20, gf24, gf32, gf64, gf128); gf4 and gf12 remain.**

## Why

The prior loop fixed GF8 and proposed two next targets; Option B (the five
mul-only rungs) was the bigger correctness win because they shared one root cause,
so a single fix repairs all five. Safe to do autonomously: the rungs were already
broken (a fix cannot regress them), the fix is exhaustively/statistically
verifiable, and it is source-only -- the Gamma tapeout (submitted 2026-05-17) is
frozen, so silicon is unaffected.

## Implementation (in tt-trinity-gamma)

- `test/gen_gf_mul_fix.py`: a one-shot generator that emits a correct `gfN_mul`
  per rung from `(E, M, bias)` and the existing special constants. The corrected
  datapath: `full_prod = {1,mant_a}*{1,mant_b}` at the full `2(M+1)` bits;
  `full_prod[2M+1] -> exp+1` else stay; mantissa = the `M` bits below the leading
  1; `guard/round/sticky` round-to-nearest ties-to-zero; mantissa-overflow and
  exp overflow/underflow handled. Regenerated `src/gf{20,24,32,64,128}_mul.v`.
- `test/gf_mul_sweep.py`: value-domain sweep for the large rungs (exhaustive is
  infeasible). Per rung, 7200 finite-normal pairs (exponents around bias, a spread
  of mantissa samples, both sign combos) through iverilog, compared in ULPs to an
  exact-rational (`fractions.Fraction`) reference. Non-circular and overflow-free.
- `docs/GF_ARITH_FINDINGS.md`: updated -- table now shows 7/9 fixed; the shared
  bug and the fix documented; remaining targets narrowed to gf4 and gf12.

## Research (this loop, web available)

The episode is a clean instance of "one root cause, many symptoms": five
independently-named units carried the identical width error because they came from
the same code-gen template that under-sized the significand product by the implicit
leading bit on each operand (`(M+1)` not `M`). A single corrected template fixes
all five and removes the divergence risk of hand-editing. The verification choice
mattered as much as the fix: at gf64/gf128 the values exceed IEEE double, so the
reference had to be exact rationals (Goldberg; Muller et al., *Handbook of
Floating-Point Arithmetic*, on the necessity of guard/round/sticky for faithful
rounding and of exact references when the format exceeds host precision). The
exponent probe alone (mantissa=0) would have passed a unit that still mis-rounded;
the value sweep is what certifies the mantissa path.

## Three Collaboration Options for the Next Loop

### Option A: Finish the ladder -- fix gf4 and gf12 (the last two)
gf4 (2-bit-ish add) and gf12 ([S1 E4 M7]) are the only rungs still broken. gf4 is
tiny (exhaustive trivially); gf12 mul is exhaustive at 2^24 pairs (tractable in
iverilog) and add is sampleable. Apply the GF8 template (proper GRS rounding).
Deliverable: gf4 + gf12 add/mul fixed and verified, `gf_arith_xcheck.py` going
7/9 -> 9/9, then wire it into CI as a green regression gate. **Recommended -- it
closes the whole arithmetic track and turns the harness into a standing gate.**

### Option B: Resolve the GF256 bias open conjecture
GF256 is excluded from the probe because its stored exponent bias does not
reconcile with `2^(E-1)-1`. Pin the actual convention from the t27 spec / decode
RTL, document it, and bring GF256 arithmetic under the same cross-check. Higher
research content (it is an open spec question), lower certainty of a clean fix.

### Option C: Audit the gfN special-value encodings
The mul fixes preserved each rung's NaN/Inf constants verbatim, but several look
inconsistent (e.g. an "Inf" constant whose mantissa field is nonzero, which the
unit's own `is_inf` test -- `mant==0` -- would classify as NaN). Cross-check every
rung's special encodings against the canonical spec and the decode path, the same
way TF3's Inf/NaN inconsistency was documented. Conformance cleanup, orthogonal to
the arithmetic datapath.
