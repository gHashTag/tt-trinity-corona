# Loop 146 Report -- 2026-06-06

## Summary

**Pivoted off the (closed) GoldenFloat track to a fresh on-silicon subsystem never
functionally checked -- `blake3_anchor`, the per-die DePIN cryptographic RECEIPT
signer -- and found a THIRD active-silicon defect, the most severe one: its BLAKE3
quarter-round G() omits ALL FOUR XOR diffusion steps. BLAKE3's G does `d = ROTR(d ^ a)`
and `b = ROTR(b ^ c)`; the RTL rotates d/b with NO prior XOR. The XOR is BLAKE3's core
nonlinear diffusion -- without it the 256-bit digest is a near-LINEAR function of the
input, so it is NOT preimage/collision-resistant, contradicting the header's "~2^96"
security claim. The RECEIPT signatures on the fabricated Gamma and Euler dies (where
blake3_anchor is the instantiated `u_hash` signer) are therefore cryptographically
weak / forgeable. Confirmed via `blake3_anchor_verify.py`: the shipped RTL == a no-XOR
model (bug present, no other transcription defect) and != the real BLAKE3-G model; the
corrected `blake3_anchor_v2` (XORs restored) == the real BLAKE3-G model. Staged v2 on
both dies; frozen on-die source untouched. Added as Defect 3 to the respin brief -- a
SECURITY defect, a stronger respin driver than the task-quiet numerical ones.**

## Research (this loop, web available)

BLAKE3's mixing function G (a refinement of ChaCha/BLAKE2) gets its security from
ARX -- Add, Rotate, Xor -- where the XOR is what makes the round non-linear and gives
avalanche; dropping it leaves Add-Rotate, which over GF(2) is close to affine and
trivially invertible, so preimage resistance collapses regardless of round count. This
is why the defect is qualitative, not a few-ULP/few-percent effect like the numerical
bugs: a hash either diffuses or it does not. It is the same archetype as the rest of
this audit's findings -- RTL that diverged from the algorithm it claims to implement,
caught by checking against the reference -- but in the cryptographic domain the
divergence is total. The lesson stands: every "compatible with X" claim deserves a
test against X.
