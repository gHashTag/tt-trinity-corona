# Loop 87 Report -- 2026-06-02

## Summary

**Independent verification of the hardest decoder (executes Loop 86 Option A). Posit8(es=0) is the most error-prone decode on the chip -- variable-length regime + terminator + fraction -- and was checked only `RTL == cocotb refmodel`, so a regime-extraction bug shared by both would pass. This loop adds a from-scratch reference (string-parses the regime run / terminator / fraction from the bit layout -- a different method than the RTL's casez priority-encoder + barrel shift, and not copied from `ref_posit8`), compiles `posit8_decode.v` with iverilog, sweeps all 256 inputs, and asserts the *silicon* output equals the independent reference. Result: 256/256 bit-exact. A mutation is caught (rc=1). This verifies the actual RTL (not a model) against an external derivation and transitively validates `ref_posit8`.**

## Why posit8 (and why iverilog)

The cocotb sweeps prove `RTL == ref_posit8`. But `ref_posit8` and the RTL were
written from the same understanding of the posit layout; a subtle regime/terminator
off-by-one could live in both. Posit regime decoding is exactly the place such a
bug hides (it is the only variable-length field in any Corona decoder). So this
loop:
- derives posit8 a third, independent way (7-bit binary string -> run-length
  regime -> k -> fraction MSB-aligned to 6 bits), and
- checks the **actual synthesizable RTL** via `iverilog -g2012` + `vvp`, not a
  Python model -- closing the loop to the bytes that ship.

## Research (this loop, web available)

Posit8(es=0): `useed = 2^(2^es) = 2`; a posit is `(-1)^S * useed^k * 2^e * (1+f)`,
with `es=0` so no exponent field. The regime is a run of identical bits after the
sign, terminated by the opposite bit; regime value `k = run-1` (run of 1s) or
`k = -run` (run of 0s); the remaining bits are the fraction. Negative posits are
the two's complement of the whole word. `0x00` is zero; `0x80` is NaR (this RTL
maps NaR -> FP32 qNaN). This matches the Posit Standard (2022) and Gustafson &
Yonemoto (2017). The from-scratch reference encodes exactly this and agrees with
the silicon on all 256 codes.

Sources: [Gustafson & Yonemoto, "Beating Floating Point at its Own Game" (2017)](https://posithub.org/docs/BeatingFloatingPoint.pdf),
[Posit Standard 2022 (posithub)](https://posithub.org/docs/posit_standard-2.pdf).
