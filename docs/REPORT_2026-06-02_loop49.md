## Research Findings

### GF180MCU Metal5 PDN (Critical Path)

- Known OpenLane issue #1549: PDN places via3 grids violating V3.2a spacing rules
- For TT GF tiles, the top-level integration owns Metal5 -- tiles must NOT use Metal5 for signals
- Our tile uses RT_MAX_LAYER=Metal4 for signal routing, but LibreLane PDN still places Metal5 power straps on 4x4 dies
- Previous attempts: FP_PDN_UPPER_LAYER=Metal4 (PDN-0003), Metal3/Metal4 PDN (PDN-0179)
- Research suggested `FP_PDN_MULTILAYER=0` but this would eliminate the power grid entirely
- wafer.space LibreLane template may have a working PDN config -- needs investigation
- **Conclusion**: This is a TT GF180MCU toolchain limitation, needs to be raised with TT team

### Post-Silicon Testing

- **tt-commander**: Web Serial API tool + Python SDK for scripted testing via RP2040 MUX controller
- For Corona ROM: script input vectors via ui_in, read uo_out, compare against golden values
- **Spacely** (Fermilab): mixed-signal framework, overkill for digital TT tiles

### D2D Between TT Tiles

- **Not feasible**: TT MUX only allows one tile active at a time, no direct tile-to-tile communication
- D2D between separate TT chips requires external wiring through demo board I/O pins
- Corona's D2D-to-Gamma plan needs external board, not on-chip routing

### Mixed-Signal on GF180MCU

- wafer.space GF180MCU Run 1: fabricated opamps, 14-bit SAR ADC, capacitive DAC (OpenFASoC)
- TT GF 0.2 has analog pin support with vertical met4 stripes for power

## Three Collaboration Options for Next Loop

### Option A: Resolve GDS Precheck with TT Team

The Metal5 PDN precheck failure is the only remaining blocker for tapeout submission. Two paths: (1) File a GitHub issue on the tt-gds-action repo explaining the contradiction -- GF180MCU 4x4 tiles require Metal5 for PDN but precheck rejects it. Reference OpenLane issue #1549. (2) Check wafer.space LibreLane template for a working GF180MCU PDN config that avoids Metal5. If neither works, ask on TT Discord whether other GF180MCU projects have the same issue and how they resolved it. This is the critical path -- without precheck passing, submission cannot proceed.

### Option B: Add NVFP4 + MXFP4 to ROM Catalog

Loop 48 research identified NVFP4 (NVIDIA Blackwell E2M1+FP8 scale, block=16) and MXFP4 (OCP MX E2M1+block scale, block=32) as high-priority new formats. Both are now production-deployed. Add them as records 80-81 via `tools/gen_rom.py`, update the SSOT t27 spec, regenerate format_rom.v, and update all tests (GLS golden for 82 records, formal ROM golden, cocotb ROM readback). This grows the catalog but does not change cell count (ROM is address-decoded). The formal prove tasks added this loop will automatically cover the expanded ROM.

### Option C: Post-Silicon Test Framework + SDC Timing

Prepare for silicon arrival (expected May 2026): (1) Create a tt-commander Python test script that exercises all 80 ROM entries and 18 decoders through the serial protocol, comparing against golden expected values. (2) Set up dual-SDC timing flow (pnr.sdc with 10% over-constraint, signoff.sdc at 25 MHz nominal) for proper timing closure. (3) Download the GDS artifact from the passing hardening run and inspect in KLayout for area/routing quality. This is the gold-standard pre-silicon preparation path and can proceed independently of the precheck blocker.
