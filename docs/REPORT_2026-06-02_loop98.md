## Assessment of the formal layer (it is sound)

The harnesses are substantive, not tautological:
- input is `(* anyconst *)` -> the SMT solver proves the property for EVERY input,
  not a sample;
- the real decoder is instantiated as `dut`;
- the core assertion is exhaustive equivalence (`assert(fp32_out == golden)` for
  computed/LUT decoders; `assert(fp32_out == {bf16_in,16'b0})` /
  `{sign,exp,mant,13'b0}` for the definitional bf16/tf32), plus structural and
  mutual-exclusion asserts and `cover` points for boundary values.

The one structural risk: the golden is hand-written and only validated when `sby`
runs. A typo'd value-table golden (nf4/fp4/posit8/lns8) would be invisible until
then. This loop closes that with a local cross-check.

## Research (this loop, web available)

Formal property verification with a golden/reference model is the strongest form
of decode verification: `anyconst` + bounded/k-induction model checking
establishes equivalence across the entire input space, not a sampled subset --
exactly what a finite combinational decoder needs. The complementary risk is the
reference itself ("who checks the golden?"), the same oracle concern as the
software layer. The defence used here mirrors the literature: an INDEPENDENT
golden (the formal harnesses even cite an external posit table, johndcook.com/
eightbit), and now a cross-layer check that the formal goldens agree with the
separately-derived software references. Five independent encodings of the decode
truth (RTL, cocotb refs, post-silicon vectors, formal goldens, Loop 86-90
references) now all agree.

Sources: [Formal verification (Wikipedia)](https://en.wikipedia.org/wiki/Formal_verification),
[SymbiYosys / property checking (YosysHQ docs)](https://yosyshq.readthedocs.io/projects/sby/en/latest/),
[8-bit posit reference table (johndcook.com/eightbit)](https://www.johndcook.com/blog/2018/04/15/eight-bit-floating-point/).
