## Research (this loop)

A useful contrast surfaced between two "XOR bind" implementations in the same
project. `holo_lut_pe` operates on BINARY hypervectors and uses bind = XOR -- which
is exactly right: in binary/bipolar MAP VSA the bind is the self-inverse XOR (or
equivalently elementwise multiply of +-1 values), so unbind = bind and the
round-trip recovers the operand. `alu9_decoder` (loop 149) operates on a TERNARY
{-1,0,+1} alphabet, where the correct bind is multiply, and its XOR-of-signed-lifts
was wrong. Same operator name, opposite correctness verdict -- because the alphabet
differs. The lesson for the audit: "is XOR the right bind?" has no context-free
answer; you must check the operand algebra the block actually works over. Both were
caught only by testing against the algebra-specific reference, not by reading the
op name.
