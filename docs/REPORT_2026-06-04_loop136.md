# Loop 136 Report -- 2026-06-04

## Summary

**Investigated whether the gf16_mul defect had a deeper "generator-level" root in the
t27 multi-language source (executes the prior loop's Option B) -- and bounded the
blast radius: it did NOT. The t27 `.t27` specs (the source compiled to 16 languages +
Verilog) define `gf16_mul` as `encode(decode(a) * decode(b))`, and `gf16_encode_f32`
does a correct round-half-up with proper mantissa-overflow handling -- so the spec has
no halving or overflow-to-zero bug. Confirmed empirically: the spec's Python reference
(`t27/conformance/gf16_ref.py`) matches the corrected RTL `gf16_v2_mul` on 99.36% of
the 262144 unit-exponent products EXACTLY, with 0 differing by more than 1 ULP; the
only differences (0.64%, all <=1 ULP) are a benign rounding-CONVENTION gap (the spec
rounds half-up; the RTL rounds ties-to-zero). Conclusion: the defect class lived
solely in the hand-written Verilog (now fixed at the t27 master, loops 134-135, and
per-die), never in the multi-language software source -- so the 16 generated language
implementations are sound. There is no buggy generator template to fix. Captured the
check as `tools/spec_vs_rtl_gf16.py`. One minor conformance nuance surfaced for future
alignment: spec round-half-up vs RTL ties-to-zero.**

## Why

The remaining "is there a deeper root?" question was whether a code-gen template fed
the bug into many targets (which would dwarf the per-die fixes). Answering it bounds
the whole arc: either there's a generator to fix, or the bug is provably Verilog-only.
It is the latter -- a strong closing result.

## Implementation

- Read the `.t27` GoldenFloat specs: per-format specs exist (gf4..gf256), and the
  Verilog in `t27/rtl_gen/` is hand-written (Makefile lists the `.v` as sources, not
  generated). `gf16_mul` in the spec = decode->f32->multiply->`gf16_encode_f32`
  (round-half-up, correct overflow) -- no manual mantissa width to get wrong.
- `tools/spec_vs_rtl_gf16.py`: compares the spec's `gf16_ref.gf16_mul` to the
  corrected RTL `gf16_v2_mul` over all 262144 unit-exp products via iverilog,
  classifying diffs as <=1 ULP (convention) vs >1 ULP (bug).

## Verification

```
spec gf16_mul vs corrected RTL gf16_v2_mul (262144 unit-exp products):
  exact: 260456 (99.36%) ; <=1 ULP rounding-convention: 1688 (0.64%) ; >1 ULP: 0
=> spec sound; bug was Verilog-only; only a round-half-up vs ties-to-zero gap
```

## Research (this loop, web available)

The result is the reassuring half of the shared-source story: a spec-first pipeline
can keep its high-level reference correct (here the spec computes in f32 and
round-encodes, which is inherently free of the fixed-width-mantissa hazard) while a
hand-written backend (the Verilog) silently diverges -- the divergence is the bug, and
it does not touch the other 15 language targets that are generated faithfully from the
same spec. The lesson is that "generated vs hand-written" is the real fault boundary:
the hand-written Verilog was never validated against the spec it was supposed to
implement, which is exactly how a halving error survives. The leftover spec<->RTL
rounding-convention gap (half-up vs ties-to-zero, <=1 ULP) is a conformance item, not
a defect -- worth aligning so the silicon and the reference agree bit-for-bit, but it
changes no result by more than one ULP.
