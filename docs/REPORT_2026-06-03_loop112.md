## Why

Loop 111 found the pinout docs advertising an unimplemented D2D interface on
`uio[7:0]` while the silicon drives the anchor byte there. That was caught by
manual audit. This guard turns the invariants it relied on into an automated
check, so any future edit that re-introduces a phantom-D2D claim, mislabels pin
directions, or changes the uio anchor-only drive without updating the docs fails
CI.

## Implementation

`test/test_pinout_consistency.py` (regex over the RTL + docs; no PyYAML):
- pin counts from `info.yaml` (8 ui + 8 uo + 8 uio);
- five RTL port-direction declarations;
- `uio_oe = is_anchor_cmd ? 0xFF : 0x00` and `uio_out_r = ANCHOR_UIO`;
- no `d2d` identifier in non-comment RTL lines (deferred);
- document-level deferral marker (`deferred` / `adr-0006`) wherever D2D is
  mentioned (info.yaml, info.md) -- robust to the marker being on an adjacent line;
- `info.md` mentions the uio anchor high byte.

## Research (this loop, web available)

The guard encodes a documentation-vs-implementation conformance check: pin
metadata must describe the as-built netlist, and a planned-but-unbuilt interface
must be explicitly marked as such. This is the same "single source of truth +
regenerate/verify" discipline applied to a place where the source (RTL ports +
the absence of D2D) and the doc (info.yaml/info.md) had silently diverged.
Automating it follows the broader practice of treating documentation as testable:
assertions over the docs, run in CI, the same way the design itself is asserted.

Sources: [Docs-as-tests / documentation testing](https://www.writethedocs.org/guide/),
[TinyTapeout info.yaml / pinout spec](https://tinytapeout.com/specifications/).
