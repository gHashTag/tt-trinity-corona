## The artifact

`docs/VERIFICATION.md` (generated, 3.4 KB) contains:
- the five evidence layers, each with the loops that built it;
- a per-decoder matrix: bits, codes, reference source (the external authority each
  independent ref encodes -- QLoRA, OCP, Posit Standard, IEEE, two's complement,
  BitNet b1.58, ...), and yes/type columns for indep-ref / formal / post-Si / mutation;
- summary counts (all derived, so they can't drift);
- the list of 17 CI gates that enforce the dossier.

It is the reviewable answer to "how do we know the silicon decodes correctly?" --
useful for the shuttle submission and any write-up, and it stays true because it
is regenerated, not hand-edited.

## Implementation

- `tools/gen_verification_matrix.py` -- imports the oracle/mutation/formal/
  post-silicon test modules, unions their `COVERED` sets, classifies each formal
  golden (LUT / computed / definitional), maps post-silicon table labels and
  aliases to decoder names, and emits deterministic markdown (sorted, no
  timestamps). `main()` returns the string; only `__main__` writes the file.
- `test/test_verification_matrix_fresh.py` -- regenerate-and-diff gate: committed
  `docs/VERIFICATION.md` must equal `gen.main()` byte-for-byte; prints the first
  differing line on failure.

## Research (this loop, web available)

A verification/traceability matrix is the standard artifact for demonstrating
that every requirement (here, every decoder's correctness) maps to concrete
verification evidence, with no "orphans" (evidence without a requirement) or
"childless parents" (a decoder with no evidence). Safety standards (ISO 26262,
DO-178C) mandate exactly this requirement->evidence traceability. The novelty
here is that the matrix is generated from the test suite and freshness-gated, so
it is always an accurate reflection of the actual checks rather than a document
that drifts -- addressing the well-known failure mode where traceability matrices
become stale paperwork.

Sources: [Requirements/verification traceability matrix](https://www.trace.space/blog/what-is-requirements-traceability),
[Traceability in functional safety (ISO 26262 overview)](https://en.wikipedia.org/wiki/ISO_26262).
