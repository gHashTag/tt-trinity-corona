## Research: TT Clock/Reset Architecture

### Reset Sequencing
- When design is **inactive**: TT mux forces `clk=0`, `rst_n=0`, all inputs=0
- When design **activates**: `rst_n` deasserts first, then `ena` goes high
- Power gating (TT06+): inactive designs have power cut; async reset flops
  initialize correctly on power-up since `rst_n` starts low

### ena Signal Contract
- `ena` is a logic gate, not a clock enable
- `else if (ena)` guard prevents state transitions during mux switchover
- Violating this risks processing garbage inputs
- Our FSM correctly implements this pattern

### GF180MCU Cell Library
- `gf180mcu_fd_sc_mcu7t5v0` includes DFF with async reset (`dffrnq`)
- OpenLane/LibreLane maps `posedge clk or negedge rst_n` correctly
- No known synthesis issues with this pattern

### Post-Silicon Clock
- Expect slower-than-simulated clocks due to scan chain through 200+ designs
- TT infrastructure adds mux delay to clock path
- 50 MHz target is plausible but may need 25-40 MHz in practice
