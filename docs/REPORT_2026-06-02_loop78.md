# Loop 78 Report -- 2026-06-02

## Summary

**Conformance-hardening loop. Web research (OCP OFP8 Rev 1.0, OCP MX v1.0, FP8 arXiv:2209.05433, E4M3FNUZ) confirmed the special-value rules every float decoder must reproduce, then audited the RTL against them. The decoders are CONFORMANT (E8M0 `0xFF`→NaN not 2^128; E4M3FNUZ `0x80`→NaN with no −0; MXFP8 E4M3 has no Inf, two NaN patterns; E5M2 IEEE-style Inf+NaN). Added an EXPLICIT, citation-backed conformance test: 22 special-value vectors across 6 formats, each cross-checked against the golden reference math (standalone, runs without cocotb) AND driven through the DUT in a new cocotb test. Fixed 3 stale decoder-count claims (README said "18 decoders"; real count is 17). cocotb tests 50 → 51.**

## Three Collaboration Options for the Next Loop

### Option A: `ml_dtypes` golden-oracle cross-check
Add a CI step that exhaustively compares each FP8 variant decoder (E4M3FN, E4M3FNUZ, E5M2) against `jax-ml/ml_dtypes` NumPy dtypes — an independent, widely-used reference implementation. Turns "our reference model agrees with our RTL" into "our RTL agrees with the industry-standard library." ~256 values × 3 formats, cheap and decisive.

### Option B: Post-silicon timing characterization (carried from Loop 77)
Extend `post_silicon/test_corona.py` to measure clock-to-output latency per decoder path on the RP2350 demoboard once silicon arrives (~Nov 2026). Would be the first published GF180MCU timing data for combinational format converters. Spec/stub now, run on silicon later.

### Option C: Formal special-value properties
Promote the 22 special-value vectors into SymbiYosys `assert` properties (e.g. "`e8m0_in==0xFF` ⇒ `fp32_out==0x7FC00000`") in the relevant `formal/fv_*.sv` harnesses. Exhaustive sweeps already cover these in sim; formal would prove them unconditionally and document conformance as machine-checked theorems.
