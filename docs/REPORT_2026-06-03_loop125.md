# Loop 125 Report -- 2026-06-03

## Summary

**Stepped back from the (now-complete) Gamma arithmetic track and did a genuine
fresh weakness scan of the whole project -- which surfaced a real cross-die issue
(executes the prior loop's Option C). The same `.t27` code-gen produced the
GoldenFloat `gfN` units for all three SKY26b dies; Gamma's were fixed over loops
119-124, but Phi and Euler ship the SAME pre-fix code. A new read-only audit tool
(`tools/cross_die_gf_audit.py`) probes each die's `gfN_add` via iverilog and
confirms: Gamma 0 buggy rungs, Phi 9, Euler 9 (gf4/8/12/20/24/32/64/128/256 -- only
gf16 correct on all three), matching Gamma's pre-fix state exactly. Crucially, the
risk is bounded: the Phi/Euler silicon top-levels instantiate ONLY the gf16 compute
path (`gf16_dot4` / `trinity_gf16_tile`), and the nine buggy units are NOT
instantiated anywhere -- standalone spec RTL never wired into the die. So the
shipped Phi/Euler chips' intended function is unaffected; the exposure is to future
reuse of the gfN library. Documented in `docs/CROSS_DIE_GF_AUDIT.md`. No die RTL was
modified (all three are frozen silicon).**

## Why

The Gamma GoldenFloat track was complete with no known remaining defect, so rather
than manufacture more Gamma work I scanned the broader project for the same class
of weakness. The TRI-NET line has four dies; three carry `gfN` arithmetic from one
code-gen. Confirming whether the Gamma bugs propagated to the siblings is a real,
high-value question -- and it does. Doing it read-only respects that Phi and Euler
are also frozen silicon (same SKY26b shuttle, submitted 2026-05-17/18).

## Implementation (in tt-trinity-corona)

- `tools/cross_die_gf_audit.py`: for each die's `src`, probes every `gfN_add` --
  `1.0+1.0` must give exponent `bias+1`, and `max_finite+max_finite` must round to a
  value that DECODES as Inf (catches the wrong Inf constant). Result decoding uses
  the correct convention, independent of each unit's (possibly wrong) EXP_MAX.
  Read-only; never touches any die's RTL.
- `docs/CROSS_DIE_GF_AUDIT.md`: the bug matrix, the not-on-compute-path risk
  assessment, and the port-Gamma's-fix recommendation for future reuse/respin.

## Research (this loop, web available)

This is a shared-generator provenance problem: one buggy code-gen fans a defect
into every consumer, so a fix in one repo does not propagate to siblings unless the
generator (or its output) is re-shipped. The audit is the mitigation -- a single
cross-die probe that any die including the family must pass before tapeout. The
saving grace here is dead-code isolation: the defective units were never
instantiated on the Phi/Euler compute path (a repo-wide instantiation search
confirms it), so the bug is latent in library RTL rather than active in silicon --
the difference between "ships wrong answers" and "would ship wrong answers if
reused." Recording that distinction explicitly is the point of the risk assessment;
silent "9 bugs found" without the not-on-compute-path qualifier would overstate the
exposure.
