# Loop 85 Report -- 2026-06-02

## Summary

**fmt_id single-source loop (extends Loop 83 from 1 consumer to all). The `FMT_<name> = id` mapping is a hardware/software shared contract (the protocol opcode on `ui_in[6:0]`) duplicated as localparams in `tt_um_trinity_corona.v` AND as Python constants across four files. Loop 83 only checked `test_decoders.py` vs the RTL; `test_stress.py`, `special_values.py`, and the Loop-84 `test_anchor_derivation.py` were unguarded -- a fmt_id renumber in any of them would silently exercise a different format than the silicon routes. New `test/test_fmt_id_consistency.py` AUTO-DISCOVERS every `FMT_<name>` constant in `test/*.py` + `tools/*.py` and asserts each matches the RTL localparam of the same name (the hardware ground truth), with no FMT_ constant the RTL lacks. Future files are covered automatically. 41 constants across 4 files verified against the 23 RTL localparams; all PASS. A drift mutation is caught (rc=1). Building the guard also surfaced and fixed a latent hex-literal parsing bug.**

`fmt_id` is the canonical "shared contract" enum (a protocol opcode), so a single
source is the correct pattern. The RTL is chosen as source of truth on purpose:
fmt_id is a hardware routing fact; changing one means changing the silicon, which
must drive the tests -- not the reverse.

## Implementation

`test/test_fmt_id_consistency.py` (stdlib, <1s):
- `rtl_fmt_ids()` -- parse `localparam [6:0] FMT_<name> = 7'd<N>` (and `7'h` for the anchor).
- `discover_consumers()` -- glob `test/*.py` + `tools/*.py` (excluding self).
- `py_fmt_ids()` -- module-level `FMT_<name> = <int>` (decimal **or** `0x` hex).
- Assert every discovered constant matches the RTL by name; flag any that the RTL
  doesn't define; require >=3 consumer files seen (so the discovery can't silently
  match nothing).

Auto-discovery means a new test file with `FMT_` constants is checked with no edit
to this guard -- unlike Loop 83's hardcoded single-file parse.

## Research (this loop, web available)

The work is a textbook **single-source-of-truth for a shared enum / identifier
drift** problem. Key points:

| Source | Finding | Bearing on Corona |
| --- | --- | --- |
| Magic-number / explanatory-constant practice | duplicated literals drift; a misspelled *named* constant is caught at compile, a wrong *number* is not | The guard is the cross-file compiler the duplicated `FMT_` constants otherwise lack |
| "shared contract" enums (opcodes, register fields, protocol codes) | a legitimate, intended single-source case -- generate from one schema, emit per target | `fmt_id` is exactly this; RTL-as-source is the pragmatic SSOT given GDS-frozen RTL |
| SystemRDL / PeakRDL / IP-XACT | one schema -> Verilog headers + C headers + test constants simultaneously | The asymptote (Loop 83 Option B): emit `fmt_ids.vh` + a Python module from the t27 catalog |

Sources: [magic numbers / explanatory constants](https://itnext.io/magic-numbers-are-problematic-5b12cfe5f31a), [enum constants over magic numbers (SSW)](https://www.ssw.com.au/rules/use-enum-constants-instead-of-magic-numbers), [single sources of truth](https://dev.to/codinsonn/the-joy-of-single-sources-of-truth-277o).
