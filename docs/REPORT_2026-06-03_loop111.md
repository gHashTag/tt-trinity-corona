# Loop 111 Report -- 2026-06-03

## Summary

**Datasheet/pinout accuracy audit (executes Loop 110 Option B) -- found and fixed a real misrepresentation. Both `info.yaml` and `docs/info.md` labelled `uio[7:0]` as "D2D TX/RX to Gamma", but the RTL has NO D2D (deferred, ADR-0006). The silicon actually drives `uio[7:0]` = the anchor high byte `0x47` during the anchor probe (`uio_oe=0xFF`) and leaves `uio` as inactive inputs otherwise. The published pin metadata therefore described an unimplemented feature and omitted the real function. Corrected `info.yaml` pin labels and header comment, fixed the `info.md` D2D prose, added an as-built pin map table, and marked the external-hardware D2D section as deferred/future. Documentation + metadata only; no RTL/test changes.**

## The misrepresentation (verified against RTL)

`tt_um_trinity_corona.v`:
- `uio_oe = is_anchor_cmd ? 8'hFF : 8'h00;` -- `uio` are outputs ONLY during the
  anchor probe, inputs otherwise.
- `uio_out = ANCHOR_UIO (0x47)` during anchor, else `0x00`.
- `uio_in` is unused (`_unused = &{uio_in, ...}`); no D2D logic anywhere.

So `uio[7:0]` carry the anchor high byte `0x47` during the `0x7F` probe and are
inactive inputs otherwise. The `d2d_tx_*/d2d_rx_*` labels were aspirational (the
deferred two-die plan), not what the shipped silicon does.

## Research (this loop, web available)

A datasheet's pinout must describe the **as-built** silicon, not aspirational
features: pins labelled for a function that the netlist does not implement are a
classic source of bring-up confusion and incorrect board designs. The fix follows
the same single-source/honesty discipline used throughout -- the pin docs now
match the RTL's actual `uio` driving (anchor byte) and explicitly defer D2D to
ADR-0006, rather than advertising an interface the chip cannot perform.

Sources: [Datasheet pinout conventions](https://en.wikipedia.org/wiki/Datasheet),
[TinyTapeout info.yaml / pinout](https://tinytapeout.com/specifications/).
