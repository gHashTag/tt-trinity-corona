## The model (protocol contract, deterministic)

A decode is `CMD1(fmt_id) + CMD2(byte_count) + N data bytes`, then the FSM streams
the 32-bit result over 4 STATUS cycles. So latency from CMD1 is purely a function
of N:

| Input | Decoders | To first result byte | To full 32-bit result |
| --- | --- | ---: | ---: |
| 1 byte | 15 (fp8/posit8/int/bcd/...) | 4 cyc | 7 cyc |
| 2 bytes | bf16 | 5 cyc | 8 cyc |
| 3 bytes | tf32 | 6 cyc | 9 cyc |

This matches `CoronaDriver.decode()` exactly (CMD1 clock #1, CMD2 #2, data #3..,
first STATUS read at #(2+N+1), last at #(2+N+4)). It is the protocol's latency
contract; on silicon the same sequence confirms the FSM advances on schedule.

## Research (this loop, web available)

Latency and Fmax are the two halves of a datasheet timing section: Fmax (Loop 108)
is the analog, process-dependent maximum clock; latency-in-cycles is the digital,
protocol-determined transaction length, which for a fixed serial protocol is exact
and known pre-silicon. Publishing the cycle-latency table now (and confirming it on
silicon) is standard datasheet practice -- the cycle count is a contract the
silicon either meets or reveals an FSM bug. Together they give "X MHz, Y cycles ->
Z ns per decode" once Fmax is measured.

Sources: [Latency vs throughput / cycle counts in datasheets](https://en.wikipedia.org/wiki/Latency_(engineering)),
[Shmoo plot / Fmax characterization](https://en.wikipedia.org/wiki/Shmoo_plot).
