# Loop 92 Report -- 2026-06-02

## Summary

**Found and fixed a real bug in the post-silicon bring-up oracle, then guarded it. `post_silicon/test_corona.py` carries hardcoded (input -> expected fp32) vectors for every on-die decoder -- the oracle that will declare the PHYSICAL chip good or bad on the RP2350 demoboard (~Nov 2026). It was hand-written separately from the RTL and 5 of the 8 `FP8_E5M2` vectors were WRONG: codes 0x7E/0x7F/0xFE/0xFF (exp=31) were labelled max-normal/Inf but are NaN per IEEE-style E5M2, and 0x01's subnormal was mis-scaled. A correct chip (the verified RTL outputs NaN there) would have been declared FAILED during bring-up. Corrected the table against the verified model (true max-normal is 0x7B=57344=0x47600000; +Inf=0x7C; NaN=0x7D) and added `test/test_post_silicon_vectors.py`, which pins all 89 bring-up vectors to the independent references from Loops 86-90. All pass; mutation caught (rc=1).**

## The bug (would have failed a good chip)

| code | old post-silicon | correct (RTL/IEEE E5M2) | why |
| --- | --- | --- | --- |
| 0x7E | 0x477F0000 "max normal" | 0x7FC00000 NaN | exp=31, m=2 -> NaN; and 0x477F0000 is 65408, not 57344 |
| 0x7F | 0x7F800000 +Inf | 0x7FC00000 NaN | exp=31, m=3 -> NaN; +Inf is 0x7C |
| 0xFE | 0xC77F0000 | 0xFFC00000 -NaN | as above |
| 0xFF | 0xFF800000 -Inf | 0xFFC00000 -NaN | as above |
| 0x01 | 0x33800000 | 0x37800000 | subnormal = 2^-16, not 2^-24 |

Corrected `FP8_E5M2_VECTORS` to meaningful, validated codes: 0x7B=max-normal
(57344=0x47600000), 0x7C=+Inf, 0xFC=-Inf, 0x7D=NaN, 0x01=min-subnormal (2^-16),
plus ±0 and 1.0. The other 16 tables (83 vectors) were already correct.

## Implementation

`test/test_post_silicon_vectors.py` (stdlib, imports the Loop 86-90 reference
modules; no iverilog):
- Maps each `*_VECTORS` table to its independent reference and applies the
  top-module output assembly (lns8 -> `{sign,15'b0,mag}`; bcd -> `{25'b0,bin7}`;
  int4/int8 -> sign-extend; others -> the decoder's fp32).
- Compares every hardcoded bring-up vector to the reference; fails on any drift.

This makes the bring-up oracle a derived consumer of the verified model rather
than a third independent (and previously wrong) hand-copy.

## Research (this loop, web available)

E5M2 follows IEEE-754 special-value conventions (unlike E4M3 and the FNUZ
variants): exponent all-ones (11111) with mantissa 00 is Inf (0x7C/0xFC), and
nonzero mantissa is NaN (0x7D-0x7F, 0xFD-0xFF); max finite magnitude is 57,344.
A documented subtlety: the OCP spec leaves the three NaN mantissa payloads
*undefined*, so a bring-up test should assert "is NaN" rather than a specific
payload -- here the RTL canonicalizes NaN to 0x7FC00000, so the exact-value
assertion is valid for THIS implementation. The original table's error was
exactly the classic confusion the research warns against: treating exp=31 codes
as finite/Inf and (separately) using a wrong subnormal scale.

Sources: [OCP OFP8 Rev 1.0](https://www.opencompute.org/documents/ocp-8-bit-floating-point-specification-ofp8-revision-1-0-2023-12-01-pdf-1),
[FP8 Formats for Deep Learning (arXiv:2209.05433)](https://arxiv.org/abs/2209.05433),
[NVIDIA FP8 intro (E5M2 range +/-57344)](https://developer.nvidia.com/blog/floating-point-8-an-introduction-to-efficient-lower-precision-ai-training/).
