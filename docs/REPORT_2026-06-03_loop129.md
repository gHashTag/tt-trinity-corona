## Research (this loop, web available)

The subtle part of the fix is that the address width and the hash width must move
together: widening `neuron_base` to 11 bits while leaving `w_gen`'s mixing function
keyed on bits [9:0] would have preserved the collision (addr 1024 and 0 hash
identically when bit 10 is ignored) -- the defect would *look* fixed by the param
width but the aliasing would persist. Verifying behaviorally (old vs v2 outputs
differ) rather than trusting the width change is what catches that. The Corona
result is the complementary point: a clean full-design lint on the primary die is a
genuine negative result worth recording -- it bounds where the bug class is (the
sibling dies' demo/library RTL, not the verified oracle) and validates that the
lint methodology's two hits were real signal, not tool noise.
