## Research (this loop)

Why gate a *characterization* test (one that asserts a bug is present) rather than
only a *fix* test? Because the fabricated dies are frozen: the shipped RTL must stay
exactly as taped out for provenance, and the corrections live only in the `_v2`
modules for a respin. A gate that asserted "blake3 is correct" would force editing
the frozen source to pass -- wrong. Instead the gate pins the known state on both
sides: shipped == the characterized-broken model, v2 == the reference. That turns
the audit into a two-sided invariant -- it fails if someone "fixes" the frozen
source (provenance break) OR regresses a v2 fix (lost correction). This is the
correct CI shape for a post-silicon audit where the silicon cannot change but the
record of what is wrong, and what the fix is, must not rot.
