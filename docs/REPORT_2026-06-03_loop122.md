## Why

The datapath was correct but I had preserved the special-value constants verbatim
during the mul fixes precisely because they were out of scope then. Auditing them
(Option B) is the completion: an arithmetic unit whose overflow emits a non-round-
trippable "Inf" is still wrong. This is the right next step over Option A
(re-touching the silicon gf16) which would break source<->silicon provenance.

## Research (this loop, web available)

Two reinforced lessons. First, a correct datapath is not a correct float unit:
IEEE-754 requires Inf/NaN to be *encoded* and *recognised* consistently, and here
the produced "Inf" failed its own `is_inf` test -- the classic round-trip property
(decode(encode(x)) == x) is the cheap check that catches it. Second, the
exact-rational reference must never materialize the bound: comparing |x| against
2^(2^(E-1)) by constructing that integer is a multi-million-digit operation for
gf64/gf128; comparing exponents via floor-log2 (and only materializing small
bounds) is the scalable form -- a reminder that arbitrary-precision code needs the
same overflow-awareness as fixed-width hardware. Sources: IEEE-754 special-value
semantics; Muller et al., *Handbook of Floating-Point Arithmetic*.
