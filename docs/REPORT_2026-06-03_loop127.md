# Loop 127 Report -- 2026-06-03

## Summary

**A fresh, broader weakness scan -- a full-design verilator lint from each die's
silicon top, beyond the GoldenFloat units I'd been focused on -- found the first and
only ACTIVE silicon arithmetic defect of the whole arc: a rounding-overflow bug in
`gf16_mul`, which IS instantiated on the gf16 MAC path (`gf16_dot4`) of all three
SKY26b dies. `mant_rounded` is declared `[8:0]` (M=9 bits, not M+1), so `mant_out+1`
wraps when the mantissa is all-ones and the overflow test `mant_rounded[9]` reads a
nonexistent bit (verilator SELRANGE, always 0). When a product mantissa rounds up
across a binade boundary the exponent is NOT incremented -> the result is exactly
HALVED. New `test/gf16_mul_silicon_bug.py` sweeps all 512x512 mantissa pairs at unit
exponent vs the corrected `gf16_v2_mul`: 189/262144 = 0.072% of products differ, ALL
exactly halved (e.g. 1.002 x 1.996 = 2.0 returns 1.0). Unlike the previously-
documented gf16 caveats (cancellation, format overflow) which the near-1.0 workload
avoids, this hits NORMAL products -- so the validated MNIST/IGLA workload does NOT
avoid it, which strengthens the respin case. The fix already exists and is verified
(`gf16_v2_mul`, correct M+1-bit width). The frozen `gf16_mul.v` is left as taped out;
the risk assessment in both findings docs is corrected.**

## Why

I had been auditing arithmetic by hand-reading the gf units; the systematic move was
a full-design lint from the actual silicon top, which elaborates only the
instantiated hierarchy and reports latches / out-of-range selects / width bugs in the
logic that is really on the chip. That surfaced the gf16_mul SELRANGE in one shot --
a defect my targeted gf16 probes (1+1, 2*1) had missed because they never hit a
mantissa that rounds up across a binade.

## Implementation

- **Scan:** `verilator --lint-only -Wall --top-module <tt_um_*>` on Gamma, Phi,
  Euler. Serious classes only. Found `SELRANGE gf16_mul.v:111 (mant_rounded[9] on
  [8:0])` on all three; plus some `WIDTHTRUNC` in vsa_matmul / popcount / blake3 /
  bitnet (triage deferred -- likely intentional low-bit masking).
- **Characterize:** `tt-trinity-gamma/test/gf16_mul_silicon_bug.py` -- exhaustive

- **Document:** `GF_ARITH_FINDINGS.md` gets the active-defect section + a REVISED
  respin risk (the bug is on-path and not avoided by the workload); the Corona
  `CROSS_DIE_GF_AUDIT.md` bottom line is corrected from "no active defect" to "one
  active defect (gf16_mul), fix = gf16_v2_mul".

## Research (this loop, web available)

The methodological lesson dominates: targeted functional probes (1+1, 2*1) and even
a value sweep that misses the fragile mantissa miss a rounding-overflow defect that a
structural lint catches instantly -- an out-of-range bit-select is a static property,
visible without any stimulus. Linting from the true top (so only instantiated logic
is elaborated) is what separates "active silicon defect" from "dead library code":
the same class of bug (mantissa width one bit short) was dead code in the gfN units
but live in gf16_mul. The defect itself is the canonical missing-guard-bit/overflow
normalization error (Goldberg; the rounded significand needs M+1 bits to detect carry
out of the MSB), and the round-half-up-to-power-of-two case is precisely where it
bites. Pairing the static finding with an exhaustive-over-mantissa dynamic
characterization turns "there is a SELRANGE" into "0.072% of products are halved",
which is what a respin decision actually needs.
